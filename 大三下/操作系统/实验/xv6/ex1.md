1）设计思路（xv6 上的 COW）

核心目标：fork() 时不复制物理页，只共享；当任一方写入时才复制该页。
页表标记
新增 PTE 标志位（例如 PTE_COW），表示“这是 COW 页”。
fork 后把父/子共享页的 PTE_W 清掉，设置 PTE_COW，变成只读。

引用计数
在 kalloc 维护每个物理页的 refcount。
fork 时对共享页 refcount++；释放页时 refcount--，为 0 才真正 kfree。

写时复制触发
当进程写共享页，会触发 store page fault。
处理流程：分配新页 -> 复制旧页内容 -> 更新 PTE 为可写、清 PTE_COW -> refcount--。

内核写用户内存
copyout() 也可能写共享页，需识别 PTE_COW，触发同样的复制逻辑。

TLB 刷新
修改页表后需要 sfence.vma 刷新 TLB。

2) 何时提升性能（分析）

有利场景：
fork 后立即 exec：子进程几乎不写父进程地址空间，COW 几乎不发生，省掉大量拷贝。
父子仅读取共享页：如只读代码段、只读数据段，共享带来更小内存占用、更少缓存污染。
短生命周期子进程：子进程很快退出，几乎不写，内存占用和复制成本都低。
不利场景：
写密集工作负载：父子都大量写内存，会触发大量 COW fault，反而比直接复制更慢。
长时间运行且内存逐渐分化：最终多数页都会被复制，额外的 fault 成本变成负担。

结论表述：
COW 的优势来自“延迟复制”，适合读多写少、fork+exec 模式、短生命周期子进程；
对于“写多、长期并发写同一地址空间”的场景，COW 会引入更多 page fault 和复制开销，收益下降。


实验结论：

通过本次实验，在Linux+Qemu环境下对xv6进行了多项内核功能扩展，完成了基础系统调用、调度器改造、信号量机制与slab内存管理等内容。整体结果表明：在xv6这样结构清晰、模块化的教学内核上进行功能扩展是可行且直观的，实验过程也加深了对进程调度、同步机制与内存管理的理解。
1.	基础系统调用与HelloWorld
成功新增系统调用并在用户态调用运行，验证了系统调用从用户态进入内核、参数传递与返回值路径，掌握了syscall.h/syscall.c/usys.pl等关键文件的协作关系。
2.	调度器实验
o	时间片扩展：通过在定时器中断中维护ticks_remaining，实现了N个tick 才触发切换的机制，并可通过内核日志验证时间片到期时的切换频率。
o	优先级调度：在proc结构中加入优先级字段，在调度器中选择最高优先级的可运行进程，实现了静态优先级调度。实验观察到高优先级进程调度频率更高，体现了调度策略对CPU占用的影响。
3.	信号量机制
实现了 sem_init/sem_wait/sem_post/sem_destroy，并用父子进程交替输出、阻塞唤醒等测试验证：
o	sem_wait 在计数为 0 时能阻塞；
o	sem_post 能正确唤醒；
o	信号量可实现互斥与同步。
4.	slab 内存管理
采用固定16页作为slab池，支持8种固定块大小，完成slab_alloc/slab_free系统调用。测试结果显示16~1024块大小能正常分配回收，超范围请求会返回失败，表明slab基础框架可用，验证了小对象分配的机制。
5.	COW设计思路与性能分析
通过设计COW机制（共享页、只读标记、写时复制、引用计数），可以在fork后减少不必要的内存复制。
结论是：COW对于“fork后立即exec、读多写少、子进程短生命周期”的场景性能提升明显；而在“写密集或长时间分化地址空间”的场景下，额外的page fault 和复制开销会降低收益。

实验加深了对 xv6 调度器实现细节、同步机制与内核内存管理机制的理解，能够掌握从需求到内核代码实现、再到用户态测试验证的完整流程，为后续深入内核机制打下基础。我将在日后的学习中不断精进自己的能力，努力将所学应用实际。
