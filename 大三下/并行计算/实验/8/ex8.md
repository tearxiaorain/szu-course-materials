一、算法逻辑与设计思路

双调排序是一种数据无关的比较排序算法，其基本思想是：首先自底向上将数组逐步构造成越来越大的双调序列（先升后降或先降后升），然后通过一系列的蝶式比较交换将双调序列归并为完全有序。算法的核心是两层嵌套循环：外层 k 控制当前双调块的大小（从 2 翻倍到 n），内层 j 控制蝶式比较的距离（从 k/2 折半到 1）。每次内层迭代比较距离为 j 的元素对 (i, i^j)，方向由 (i & k) == 0 决定——k-bit 为 0 的元素参与升序比较、为 1 的参与降序比较，从而相邻块自动形成交替升降序的双调序列。

在具体实现中，采用三层优化策略来充分利用 GPU 的内存层次结构：

（1）全局内存阶段（bitonic_step）：当比较距离 j ≥ TILE（TILE=16384）时，元素对跨越多个线程块，必须在全局内存中进行交换。使用 Insert_zero 位映射技术，将 n/2 个线程一对一映射到所有"第 log2(j) 位为 0"的元素下标，每个线程恰好处理一个比较对，达到 100% 的线程利用率。

（2）一次性共享内存初始排序（bitonic_init_shared）：由于前 14 层（k=2 到 k=16384）的所有比较对都在同一个 TILE 内部，本程序将这一阶段合并为一次核函数调用。每个线程块使用 64KB 动态共享内存加载 16384 个元素（grid-stride 方式，每线程加载 16 个元素），在共享内存中完成全部 k=2..16384 的所有 j 循环，大幅减少了核函数启动次数和全局内存访问。

（3）共享内存合并阶段（bitonic_merge_shared）：对于 k > TILE 的每个阶段，先通过全局内存核函数完成 j ≥ TILE 的跨块交换，再启动一次共享内存核函数，将 j < TILE 的剩余阶段合并完成。init_shared 和 merge_shared 的区别在于前者包含外层的 k 循环而后者只做固定的 j 循环，两者均使用 pair-index 映射（i = 2p - (p & (j-1))）实现每线程处理 8 个比较对。

二、实验过程中遇到的问题及解决方法

（1）n/2 线程直映射导致排序错误：最初使用"线程号即元素下标"的直映射方式，虽然检查了 partner > i 来避免重复操作，但当 j < n/2 时，部分比较对的双方元素下标均大于 n/2，n/2 个线程无法覆盖这些比较对，导致排序结果错误。解决：借鉴 PPT 代码中的 Insert_zero 映射技术，通过在线程号的第 log2(j) 位插入一个 0，将线程号精确映射到第 log2(j) 位为 0 的元素下标集合，恰好覆盖所有 n/2 个比较对。

（2）共享内存跨块访问错误：在将 TILE 扩展为 16384 的共享内存核函数中，最初使用 partner = tid ^ j 计算搭档在共享内存中的下标。当 j = 1024 时，tid 与 tid^1024 恰好跨越两个相邻的线程块（距离 = 1024），共享内存无法访问跨块数据。解决：将共享内存核函数的 j 上限从 1024 调整为 512（即 j < TILE/2），使所有比较对严格限制在同一线程块的 1024 个元素范围内。

（3）Warp Shuffle 越界：在尝试对 j ≤ 32 阶段使用 __shfl_xor_sync 进行寄存器级交换时，最初将阈值设为 32，但 warp 只有 32 个 lane（lane ID 0-31），j=32 时 XOR 结果可能超出有效 lane 范围。解决：将 warp shuffle 阈值下调为 j ≤ 16，j=32 改走共享内存路径加 __syncthreads()。

（4）int 溢出导致死循环：主机端循环 for (int k = TILE << 1; k <= n; k <<= 1) 中，当 n = 2^30 时，k 在最后一轮迭代后翻倍为 2^31，超出 int32 最大值 2,147,483,647 而溢出为负数，循环条件 k <= n 永远为真，程序卡死。解决：将 k 的类型改为 long long。

三、实验结果分析

各问题规模下的 CPU/GPU 执行时间和加速比的具体数据如表 1 所示。

CPU 执行时间从 2^23 时的 0.593s 增长到 2^30 时的 99.650s，大致呈现随规模线性增长的趋势（n 每翻倍，排序时间约翻倍），符合排序算法 O(n log n) 的理论复杂度。

GPU 执行时间从 2^23 时的 0.002s 增长到 2^30 时的 1.120s。在 n 从 2^23 增长到 2^24 时，GPU 时间仅从 2.0ms 增加到 4.2ms（约 2 倍）；但当 n 从 2^24 进入 2^25 时，GPU 时间从 4.2ms 跃升至 19.2ms（约 4.6 倍），此后每个规模的增幅维持在约 2.2-2.3 倍。这一阶段性变化的原因是：当 n ≤ 2^24 时，n ≤ TILE，整个排序过程完全由一次 bitonic_init_shared 核函数在共享内存中完成，无任何全局内存交换和额外的核函数启动，因此效率极高、时间极短；当 n ≥ 2^25 时，排序需要跨 TILE 的全局内存交换（bitonic_step），核函数启动次数和全局内存访问量大幅增加，导致 GPU 时间增长斜率变大。

加速比方面，整体呈现"先高后低"的趋势：在 n = 2^23 和 2^24 时加速比达到最大值 295.57（GPU 比 CPU 快近 300 倍），这是由于 n ≤ TILE 阶段全在共享内存完成，GPU 的大规模并行和高速片上内存优势得到最充分发挥；从 2^25 开始加速比逐步下降，从 133.32 持续降至 2^30 时的 88.99。加速比下降的主要原因有两点：第一，当 n > TILE 后，引入了大量跨块全局内存交换的核函数，每次核函数启动都有固定开销，且全局内存带宽远低于共享内存，存储墙效应逐渐显现；第二，双调排序的总比较次数约为 (log2 n × (log2 n + 1)) / 2 轮，核函数启动次数随 n 增加而增长（从 2^25 时的约 120 次到 2^30 时的约 135 次），固定的核函数调度开销占比随规模增大而上升。但即便如此，GPU 在所有测试规模下相对于 CPU 均有 88 倍以上的加速比，体现了 GPU 在数据并行密集型任务上的显著优势。

与上次矩阵乘法实验对比，矩阵乘法的加速比随规模增大而持续上升（计算密集型，O(n^3) vs O(1) 通信），而双调排序的加速比随规模增大而缓慢下降（访存密集型，O(n log n) 计算 vs O(n) 访存，且核函数启动次数随 log2(n) 递增）。这一差异恰好体现了不同类型并行算法在 GPU 上的性能特征：计算密集型的收益随规模单调递增，而访存和同步密集型在大规模下会受到存储墙的限制。

四、心得体会

本次实验我完成了基于 CUDA 的并行双调排序算法，依次实现了从最简全局内存版到 Insert_zero 线程映射、共享内存合并、warp shuffle 等多层次优化版本，深刻体会到了 GPU 内存层次结构对排序算法性能的决定性影响。在调试过程中，遇到了线程映射遗漏、共享内存跨块访问、warp shuffle 越界、int 溢出死循环等一系列典型问题，每次排查都加深了我对 CUDA 线程模型和硬件限制的理解。通过测试不同规模下的执行时间和加速比，我认识到即使在所有规模下 GPU 都远超 CPU 的排序任务中，加速比也不是恒定的——全共享内存的小规模阶段加速比最高，引入全局内存交换后加速比逐步回落，这体现了"片上内存 vs 片外内存"的巨大性能鸿沟。在日后的学习中，我将不断精进自己的能力，努力将理论知识与实际应用相结合。
