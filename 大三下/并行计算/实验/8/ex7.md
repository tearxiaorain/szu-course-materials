五、实验总结
实验涉及的关键知识点、算法逻辑或设计思路；实验过程中遇到的问题、原因及解决方法；实验结果的分析。

本次实验使用 CUDA 实现了两个 n 阶方阵的并行乘法，并对结果矩阵所有元素求和（使用线程束洗牌），与 CPU 串行结果进行正确性验证。通过对不同问题规模 n 的测试，分析并行程序的执行时间和加速比。

在该次实验中两个CUDA核函数的算法逻辑和设计思路如下：
Matrix_mult 核函数 —— 矩阵并行乘法：
采用"一个线程计算一个结果元素"的并行策略。将线程组织为二维网格与二维块，每个线程通过 blockIdx 和 threadIdx 计算其负责的全局行列索引：
"row"="blockIdx.y"×"blockDim.y"+"threadIdx.y" 
"col"="blockIdx.x"×"blockDim.x"+"threadIdx.x" 
线程执行点积计算 z["row"]["col"]=∑_(k=0)^(n-1)▒〖x["row" ][k]×y[k]["col" ]〗。
在此基础上引入共享内存分块（Tiling）优化：以 TILE_WIDTH=16 为单位，块内 256 个线程协同将 x 和 y 的 16×16 子块从全局内存加载到共享内存中，然后从高速共享内存读取数据完成局部乘加。每轮加载前后同步，确保数据一致性。

Sum 核函数 —— 并行归约求和：
采用三级归约策略，使用线程束洗牌完成核心归约：
线程级局部求和：使用网格跨步循环，每个线程负责跳跃累加多个元素，得到局部和。
Warp 内归约：使用 __shfl_down_sync 指令，在一个Warp内通过offset递减完成归约，结果留在每个Warp的第一个线程中。
Block 内归约：各Warp的lane 0将部分和写入共享内存数组shared_sums，块内第一个Warp再次使用线程束洗牌对这些Warp和进行归约。
Grid 级归约：每个块的第一个线程使用atomicAdd将块总和原子累加到全局结果sum_p中，避免多块并发写入冲突。

实验过程中遇到的问题及解决方法：
线程数不足导致结果不完整：如果启动的线程总数不足以覆盖所有 n^2 个元素，未分配线程的位置将保持未初始化状态。
解决：使用向上取整公式 (n + TILE_WIDTH - 1) / TILE_WIDTH 计算网格大小，确保覆盖全矩阵；核函数内部加入 if (row < n && col < n) 边界检查，让越界线程直接退出不做计算。

原因分析：
在同等矩阵阶数下，问题规模越大，CPU 执行时间增长越剧烈，这是由矩阵乘法 O(n^3) 的计算量决定的。而 GPU 执行时间始终保持在毫秒级且增长平缓，说明 GPU 的大规模并行架构能够有效消化计算量的增长。
n≤128 时加速比小于 1，主要原因是GPU存在固定的启动与通信开销：CUDA核函数启动、主机到设备的两次数据传输、设备到主机的结果回传、以及同步操作。当 n 较小时，实际的并行计算时间非常短，这些固定开销占总执行时间的绝大部分，导致GPU总时间远超CPU串行执行。此外，小规模下 n^2 个线程不足以占满 GPU 的全部流式多处理器，硬件利用率低也加剧了性能劣势。
n=256 时加速比突破1，说明GPU并行收益开始超过开销成本。此后加速比随 n 增大持续攀升，这是因为：n 增大意味着 n^2 个线程的计算量按 O(n^3) 增长，而固定开销基本不变，开销占比快速下降；同时线程数增多使得GPU各SM的占用率提高，能更好地通过线程调度隐藏内存访问延迟；此外共享内存分块优化在更大规模下效果更显著——每个线程块的 16×16 共享内存子块被更充分地复用，全局内存访问占比进一步降低。因此加速比随 n 增大而快速提升。
总体而言，本实验体现了 GPU 在计算密集型、高度可并行的矩阵运算中"小规模不划算、大规模显优势"的典型特征，只有当 n 超过一定阈值后，并行化的收益才能显著超越开销。
